Building an AArch64 OS Kernel
from Scratch in Rust

A from-scratch kernel on QEMU's ARM virt machine. The goal isn't just to build something that boots — it's to understand why the hardware is the way it is.

⊕ North Star — Hardware/Software Co-Design Every implementation milestone maps to a hardware question. When you write a TLB invalidation, you should know why TLBs exist and why they're expensive on multicore. When you write a spinlock, you should understand why ARM's memory model is weaker than x86's — and what that costs you. The OS is the hardware's first client.
⚒ Ground Rule — Everything By Hand No helper crates: no aarch64-cpu, no tock-registers, no fdt, no elf parsers. Every MSR/MRS is inline asm you wrote; every MMIO register is a write_volatile at an address you verified in the DTB; every on-disk struct is a #[repr(C)] you laid out. Slower — and exactly the point.

Hardware Deep Dive
⚡ MMIO, UART, and the QEMU virt machine

Why UART? UART is used universally for early OS debug because it requires no driver stack — it's a direct memory-mapped register. Write one byte to one address; the hardware serializes it. No interrupts, no DMA, no buffer management. This is why every OS tutorial in history starts here.

MMIO vs. Port I/O: x86 has two ways to talk to devices: port I/O (in/out instructions, a separate 64KB address space) and MMIO (devices appear as ordinary memory addresses). ARM has only MMIO — a deliberate ISA design choice that simplifies the bus architecture and allows normal load/store instructions to reach peripherals. This is why AArch64 has no dedicated I/O instructions.

QEMU virt machine memory map (simplified): 0x0000_0000–0x0800_0000 Flash / ROM QEMU firmware 0x0800_0000 GICv2 interrupt controller (Phase 5) 0x0900_0000 PL011 UART ← your console 0x4000_0000–end of RAM DRAM ← kernel loads here

Which EL does QEMU start you at? Real ARM hardware boots at EL3 (Secure Monitor), drops through EL2, and hands the OS control at EL1. Plain -machine virt skips all of that and starts your kernel directly at EL1 — convenient, but you'd never touch the privileged boot path. This plan boots with -machine virt,virtualization=on, which starts you at EL2 so Phase 1 makes you perform the EL2 → EL1 drop yourself, exactly as real boot firmware does. Verify with CurrentEL as your first instruction.

Hardware Deep Dive
⚡ Why Exception Levels Exist — The Hardware Argument

The hardware motivation: Without privilege separation, any user program could write to kernel memory, reconfigure the MMU, or disable interrupts. The CPU enforces isolation in silicon. This isn't policy — it's physics. The ISA defines which registers and instructions are accessible at each privilege level, and violations trap synchronously.

AArch64 Exception Level Stack: ┌──────────────────────────────────────────────────┐ │ EL3 Secure Monitor — TrustZone / Secure Enclave │ ← Not used in this project │ EL2 Hypervisor │ ← QEMU starts you here │ EL1 OS Kernel │ ← YOUR kernel lives here │ EL0 User Applications │ ← Added in Phase 6 └──────────────────────────────────────────────────┘ Higher EL = more privilege = more hardware access

AArch64 vs. x86: x86 defined four protection rings (0–3) but modern OSes use only ring 0 (kernel) and ring 3 (user). ARM's design is cleaner: EL0–EL3 are explicitly defined for user/kernel/hypervisor/secure-monitor. ARM added a dedicated hypervisor mode (Hyp in ARMv7-A's Virtualization Extensions, which became EL2 in ARMv8) specifically because virtualization without hardware support was a performance disaster — the OS had to trap every privileged instruction. x86 added VMX later, and it's still baroque by comparison.

TrustZone and Apple's Secure Enclave: EL3 plus the Secure world is TrustZone — ARM's standard mechanism for code that even the kernel cannot touch (boot firmware, key storage, DRM). Apple's SEP, notably, is not TrustZone: it's a physically separate coprocessor with its own CPU, SRAM, and boot ROM. Apple chose discrete silicon over a privilege level because a separate processor stays isolated even if EL3 firmware on the application cores is compromised. The lesson is the one this project keeps teaching: isolation guarantees come from hardware design, and there's a spectrum — privilege levels, TrustZone worlds, separate silicon — each trading cost for assurance.

PSTATE: AArch64 exposes the current CPU state as PSTATE — a collection of flags: N/Z/C/V (condition codes), DAIF (interrupt masks), SPSel (stack pointer select), and the current EL. Saving and restoring PSTATE correctly is what makes a context switch transparent to a task. Phase 6 will save it explicitly.

Hardware Deep Dive
⚡ Why 4KB Pages? Granule Size, Cache Lines, and Apple's 16KB Choice

The 4KB page is not fundamental — it's a tradeoff. AArch64 supports three granule sizes: 4KB, 16KB, and 64KB. The granule controls page table tree depth, TLB coverage per entry, and internal fragmentation per allocation.

Granule size tradeoffs (48-bit VA, AArch64): Granule Table levels TLB coverage/entry Internal frag (1B alloc) 4 KB 4 4 KB up to 4,095 B wasted 16 KB 3 16 KB up to 16,383 B wasted ← Apple Silicon 64 KB 3 64 KB up to 65,535 B wasted

Apple Silicon uses 16KB pages — this is why macOS/iOS have 16KB alignment requirements. Larger pages reduce TLB pressure (fewer entries needed to cover the same virtual range), which matters enormously when processes have large address spaces (Xcode, Final Cut Pro). The tradeoff is higher internal fragmentation, but Apple's memory allocators are tuned for it. This is a direct example of a microarchitectural decision (large unified memory) driving an ISA configuration choice.

Cache lines and false sharing: Cache lines on AArch64 are 64 bytes. If your bitmap's byte covers 8 frames, and those frames are allocated/freed by different CPU cores, all those operations hit the same cache line — this is called false sharing. It generates spurious coherency traffic even though the cores are touching logically independent data. Linux's per-CPU page allocator is explicitly designed to avoid this. Even in single-core QEMU, understanding this prepares you for Phase 7.

Physical address space: Your QEMU virt machine has a 40-bit PA space (1TB). Real Apple Silicon (M1/M2) extends to 52-bit PA (FEAT_LPA2). The extra PA bits exist because unified memory architectures — where CPU and GPU share the same DRAM — need full address coverage across all agents. That's an example of a product architecture decision (unified memory) driving an ISA extension.

Hardware Deep Dive
⚡ Translation Table Walk Engine, TLBs, ASID, and the TTBR0/1 Split

Why does hardware walk the page table? On most modern ISAs, the CPU has a dedicated hardware state machine called the Translation Table Walk Engine (TTWE). On a TLB miss, the TTWE traverses the multi-level page table in memory — no OS involvement required. This is a deliberate choice: software-managed TLBs (like MIPS) let the OS control every miss but add kernel path length and latency. ARM chose hardware walks for predictable performance. The TTWE is a real silicon block that costs area and power — its existence is a hardware investment justified by OS workloads.

AArch64 4-level address translation (4KB granule, 48-bit VA): VA bits: [47:39] [38:30] [29:21] [20:12] [11:0] L0 idx L1 idx L2 idx L3 idx page offset (9 bits) (9 bits) (9 bits) (9 bits) (12 bits) TTBR1_EL1 (VA[63:48] = 0xFFFF) → kernel mappings TTBR0_EL1 (VA[63:48] = 0x0000) → user process mappings

The TTBR0/TTBR1 split is an AArch64 innovation. x86 has one CR3 register — switching between user and kernel requires either a full TLB flush or PCID (Process Context Identifier) tricks bolted on later. AArch64 hardwires the split: addresses starting 0xFFFF… use TTBR1 (kernel), addresses starting 0x0000… use TTBR0 (user). On a context switch, only TTBR0 changes — kernel mappings in TTBR1 are always valid. This directly reduces TLB flush costs and is a case of the ISA being designed with OS context switching semantics in mind.

TLBs and ASID: The TLB caches virtual→physical translations. Each core has its own L1 TLB (typically 48–64 entries for instructions, 32–48 for data). Without tags, every context switch requires a full TLB flush — expensive. AArch64 tags TLB entries with an ASID (Address Space ID), so entries from different processes coexist. On context switch, only TTBR0 and the ASID change; stale entries from other processes are ignored. The ASID width (8 or 16 bits, set in TCR_EL1) bounds how many process TLB entries can coexist before an ASID rollover forces a flush.

Memory attributes and MAIR: Every page table entry contains a 3-bit index into MAIR_EL1 (Memory Attribute Indirection Register), which holds up to 8 attribute profiles. This indirection lets you change attributes for all pages of a type by modifying one register. The attributes control cacheability, shareability (inner-shareable for SMP, outer for full system), and device memory strictness (nGnRnE = no gathering, no reordering, no early-write-acknowledgement — the strictest device type, used for UART). The CPU uses these to decide whether to speculatively prefetch, write-combine, or reorder accesses to that region.

Hardware Deep Dive
⚡ Why Allocator Design Is a Hardware Problem

Allocation policy decides cache behavior. Two objects allocated together are usually accessed together. A free-list allocator that scatters successive allocations across the heap turns sequential access patterns into cache-miss storms; a bump or slab allocator keeps them on the same cache lines. This is why high-performance allocators (jemalloc, mimalloc) obsess over locality, not just speed of the alloc call itself.

Alignment is a hardware contract, not a style preference. AArch64 atomic and exclusive instructions (LDXR/STXR, Phase 7) fault or lose their guarantees on unaligned addresses, and unaligned accesses that cross a cache-line boundary cost two memory transactions. Layout is Rust encoding this hardware contract in the type system — your allocator is where the contract gets honored or violated.

Why a slab allocator exists (Bonwick, Sun 1994): Free list: alloc(64) → search list → split block → write header objects scattered, headers pollute cache lines Slab: one page = array of same-size slots + bitmap alloc is "find free bit" — O(1), dense, cache-hot object constructors run once per slot, not per alloc

False sharing returns. Phase 2 introduced false sharing in the frame bitmap; the heap has the same disease. If your allocator hands two cores objects that share a 64-byte cache line, every write by one core invalidates the other's cached copy (the MESI traffic you'll study in Phase 7). This is why per-CPU allocator caches exist in Linux and every serious malloc — and why your slab stretch goal should align slots to cache lines.

Hardware Deep Dive
⚡ Interrupt Controllers, Timer Architecture, and IRQ Masking

Why a dedicated interrupt controller? A single interrupt pin worked for 8-bit microcontrollers with a handful of peripherals. Modern SoCs have hundreds of interrupt sources — UART, PCIe, USB, GPU, storage, thermal sensors — that need priority levels, routing to specific cores, per-source masking, pending state tracking, and EOI acknowledgement signaling. The GIC (Generic Interrupt Controller) is a dedicated peripheral that manages all of this and presents a clean, consistent interface to CPU cores. Apple Silicon uses an Apple-proprietary AIC instead of the ARM standard GIC — a product choice justified by performance and integration, not ISA requirements.

GICv2 interrupt categories: SGI (Software Generated Interrupts) INTID 0–15 For IPI (inter-processor, Phase 7 SMP) PPI (Per-Processor Interrupts) INTID 16–31 Timer lives here (INTID 30) SPI (Shared Peripheral Interrupts) INTID 32+ UART, PCIe, storage, etc.

The ARM Generic Timer is in the CPU core — not a peripheral. Unlike x86's PIT or HPET (bus-attached peripherals), the ARM Generic Timer is instantiated inside each CPU core. This design eliminates bus latency for timer reads and enables precise per-core scheduling without shared-peripheral contention. It counts at a fixed system frequency (CNTFRQ_EL0) and provides separate EL1 physical, EL1 virtual, EL2, and EL3 timers. That four-way split exists precisely because each privilege level needs an independent, tamper-proof timer that software running at a lower EL cannot reprogram.

DAIF flags: PSTATE.DAIF masks four exception types: Debug, SError (system errors like uncorrectable ECC), IRQ, and FIQ. Clearing DAIF.I unmasks hardware interrupts. The OS must mask IRQs when modifying shared kernel data structures — this is the hardware foundation of uniprocessor synchronization. On a single-core kernel, disable_irq() + enable_irq() is often sufficient. SMP (Phase 7 stretch) requires real locks on top of interrupt masking, because other cores ignore your DAIF flag entirely.

Hardware Deep Dive
⚡ The True Cost of a Context Switch — Beyond Register Saves

The register save is the cheap part. Saving x0–x30 + FPSIMD costs ~50 ns. The expensive part is what happens to the hardware state you can't save and restore.

What actually changes on a context switch (AArch64): Software-saved (explicit): x0–x30 (248 B), SP, ELR, SPSR, FPSIMD (~768 B total) Hardware state — not saved, implicit cost: L1/L2 caches — process code/data goes cold (capacity eviction) Branch predictor — BTB, BHB, RSB contain trained predictions for old task TLB — ASID mitigates, but capacity pressure remains Store/load bufs — pipeline fully drains on ERET ROB / rename — out-of-order execution state is discarded

The branch predictor is a security concern. The Branch Target Buffer (BTB) and Branch History Buffer (BHB) contain predictions trained by the previous process. Spectre v2 exploits this: a malicious process poisons the BTB so the kernel mispredicts into attacker-controlled code during a speculative window. This is why Linux added IBRS/eIBRS mitigations on x86, and why ARM added branch-history clearing (the CLRBHB instruction plus firmware workarounds) along with the CSV2/CSV3 ID-register fields — not instructions, but architectural guarantees an OS can query to learn whether a core needs mitigation at all. Context switching is no longer just about registers — it has security semantics baked into the ISA.

Apple Silicon omits SMT (hyperthreading) — partially for security. With Intel's SMT, two hardware threads share one physical core's execution units, caches, and TLB. When the OS schedules two processes on the same physical core, they can observe each other's cache timing — the root of Flush+Reload and other cache side-channel attacks. Apple explicitly does not implement SMT. This reduces single-threaded throughput per watt at the margin, but eliminates an entire class of hardware side-channel attacks.

Why SPARC invented register windows — and why it lost. SPARC had hardware register windows: procedure calls slid the window rather than explicitly saving registers, making call/return almost free. Context switching was hardware-accelerated. But when OS workloads grew in complexity, a full window save (the context switch case) was actually slower than software-managed Reduced ISA register saves on a pipelined core. The lesson: optimizing hardware for one workload pattern (procedure calls) can create pathological behavior in another (preemptive multitasking).

Hardware Deep Dive
⚡ Weak Memory Models, MESI Cache Coherency, and Why ARM Is Not x86

x86 has Total Store Order (TSO). ARM does not. On x86, stores appear in program order to all processors — the ISA guarantees this. On AArch64, the memory model is weak: stores and loads can be reordered by the CPU as long as the reordering is invisible to the local core. This means ARM software must explicitly insert barriers to enforce ordering for memory operations visible to other cores. This isn't a bug — it's a deliberate ISA choice that allows aggressive out-of-order execution and store coalescing on multicore designs.

AArch64 memory barrier instructions: DMB ISH Data Memory Barrier, Inner Shareable Orders memory accesses before/after the barrier Used in acquire/release of spinlocks DSB ISH Data Synchronization Barrier Stronger: stalls execution until barrier completes Required before/after MMU enable, cache maintenance ISB Instruction Synchronization Barrier Flushes pipeline + prefetch; reloads instructions from memory Required after writing SCTLR_EL1, TTBR changes Rust Ordering → AArch64 instruction mapping: Acquire load → LDAR (Load-Acquire) barrier after load Release store → STLR (Store-Release) barrier before store SeqCst → LDAR + STLR + DMB ISH

The MESI cache coherency protocol: Each cache line is in one of four states: Modified (dirty, owned exclusively), Exclusive (clean, owned exclusively), Shared (clean, other caches may have a copy), or Invalid. When two cores simultaneously try to acquire the same spinlock, the lock's cache line bounces: Core A holds it in M state, Core B requests it, A writes back (→ I), B acquires E state. This is called cache line bouncing, and it's why contended spinlocks are expensive — not the atomic instruction itself, but the coherency bus traffic it generates.

WFE/SEV as hardware hints: Spinning on a lock in a plain loop wastes power and generates continuous coherency traffic (repeated LDXR requests to the cache controller). AArch64 provides WFE (Wait For Event): the core enters a low-power state and wakes when another core issues SEV (Send Event) or an interrupt fires. A WFE-based spinlock consumes far less power and generates minimal bus traffic. The SEVL (Send Event Local) instruction primes the local event register before the WFE loop to close a race where the lock is released between the check and the WFE.

LL/SC vs. CAS — and why RISC ISAs prefer LL/SC: AArch64 uses Load-Exclusive/Store-Exclusive (LDXR/STXR) instead of Compare-And-Swap (CAS) as x86 uses. LL/SC avoids the ABA problem: between LDXR and STXR, if any core writes to the address — even if the value returns to the original — STXR fails. CAS would silently succeed, causing subtle bugs. ARMv8.1-A added CAS instructions (FEAT_LSE) for throughput on highly contended locks in datacenter workloads, but LL/SC remains the correct primitive for building new synchronization mechanisms.

⚡ DMA, Virtio, and Why Devices Bypass the CPU

Why DMA exists: Copying a 4KB sector through the CPU one register at a time (programmed I/O) burns thousands of cycles the CPU could spend elsewhere. DMA inverts the relationship: the OS hands the device a physical address, the device's own engine moves the data, and an interrupt announces completion. The descriptor rings you build this phase are exactly that handoff — your virtqueue entries hold physical addresses because the device doesn't go through your MMU.

That bypass is also the danger. A device doing DMA can write anywhere in physical memory — the MMU's protections don't apply to it. This is why real systems add an IOMMU (ARM's SMMU): a second MMU that translates and restricts device addresses, so a buggy or malicious peripheral can't scribble over the kernel. QEMU's virtio is trusted, so you skip this — but know which protection you're living without.

Why virtio instead of emulating a real SATA controller: Emulated real HW: guest writes each register → VM exit → QEMU emulates → resume (thousands of exits/op) virtio: guest fills a ring in shared memory, writes ONE doorbell register → one exit → QEMU drains the whole ring Paravirtualization = designing the "hardware" interface around what's actually expensive: the guest/host transition.

DMA vs. caches — the trap QEMU hides from you. On real hardware, the CPU cache and DRAM can disagree: if your dirty descriptor still sits in L1 when you ring the doorbell, the device reads stale DRAM. Real drivers do cache clean/invalidate maintenance (DC CVAC/DC IVAC) or map DMA buffers non-cacheable. QEMU's device model is cache-coherent so you won't hit this — but the DMB you must still place before the doorbell write (so the device observes a fully written ring) is the same ordering discipline from Phase 7, now between CPU and device instead of core and core.

Why filesystems are block-structured: Storage hardware doesn't sell bytes — flash erases in large blocks, disks read whole sectors, and the ECC protecting your data is computed per sector. The 512-byte transfer granularity propagates upward into every FS design decision: block-aligned inodes, bitmap-per-block allocation, and the buffer cache. The FS layout you design is downstream of the physics of the medium.

⚡ W^X, PAN/PXN, and Permission Faults as a Feature

W^X is enforced by page table bits you already know. Each AArch64 descriptor carries AP[2:1] (read/write per EL), UXN (Unprivileged eXecute Never), and PXN (Privileged eXecute Never). W^X — no page both writable and executable — kills the classic exploit of writing shellcode into a buffer and jumping to it. Your exec derives these bits from ELF p_flags; getting them wrong is the difference between a kernel and a vulnerability.

PXN and PAN: the hardware distrusts the kernel, on purpose. PXN ensures the kernel can never execute user pages even if tricked into jumping there (kills ret2usr attacks). PAN (Privileged Access Never, ARMv8.1) goes further: the kernel can't even read or write user memory through stray pointers — it must explicitly toggle PAN around intentional user copies. Both exist because decades of CVEs proved "the kernel is privileged" should not mean "the kernel touches anything by default." Set PXN on every user mapping; try PAN as a bonus.

Copy-on-write: a permission fault used as a feature: fork(): copy page TABLES only, mark every page read-only parent + child share all frames (microseconds) child writes a page: → permission fault → kernel sees the COW marker → allocates a frame, copies 4KB, remaps writable → returns; the store retries, succeeds Cost is deferred to pages actually written — exec() right after fork() copies almost nothing.

Why the kernel must validate user pointers: a syscall arg like sys_write(fd, ptr, len) is attacker-controlled. If the kernel blindly dereferences ptr, EL0 can pass a kernel address and use the kernel's own privilege to read or corrupt itself — the confused-deputy problem. Your check ("is [ptr, ptr+len) entirely in this task's user range and mapped?") is the software half of the contract; PAN is the hardware backstop for the cases you miss.
